## 2. 왜 이 서비스를 만드는가

기존 Daker(경쟁 서비스)의 구조적 문제에서 출발합니다:

- **로그인 장벽**: 갤러리를 보려면 로그인이 필요 → 외부인 진입 포기
- **갤러리·순위 분리**: 갤러리 탭과 수상 결과 게시판이 별도 → 왔다갔다해야 파악 가능
- **설명 없는 나열**: 카드에 설명이 없어 데모 링크 직접 클릭해야만 내용 파악 가능
- **공유 수단 없음**: 대회 결과를 외부에 공유할 URL이 없음

**우리의 답**:
- 로그인 없이 전체 열람 가능
- 순위 + 작품 + AI 요약이 한 카드에
- 대회 결과를 URL 하나로 공유 (Recap 페이지)

---

## 3. 핵심 설계 원칙

1. **보는 것에는 마찰이 없어야 한다** — 쇼케이스·Recap은 로그인 없이 접근 가능
2. **AI 요약은 더미 데이터** — 실제 Claude API 호출 없음. 프론트 키 노출·토큰 비용 문제로 seed 데이터 사용, 생성·고정 플로우만 시연
3. **AI 요약은 결정론적** — 제출 시 1회 생성·고정, 모든 관람자에게 동일한 설명 제공
4. **별도 백엔드 없음** — Next.js SSG + JSON 파일 + localStorage만으로 동작

## 8. 주요 결정 히스토리

| 결정 사항 | 선택 | 이유 |
|-----------|------|------|
| 서비스명 | STAGE | "결과물을 무대에 올린다"는 의미, Daker 클론 이미지 탈피 |
| 팀 모집 경로 | `/camp` | 제공된 명세 이미지 기준. 우리 기획서엔 `/teams`로 표기됐으나 이미지 우선 |
| 랭킹 페이지 | `/rankings` 추가 | 제공된 명세 이미지에 포함된 필수 페이지. 더미 데이터로 구현 |
| 해커톤 상세 탭 | 8개로 확장 | 이미지 기준: 개요/팀/평가/상금/안내/일정/제출/리더보드 |
| localStorage 범위 | 4가지로 확장 | 이미지 기준: hackathons·teams·submissions·leaderboards seed |
| AI 요약 방식 | 더미 seed 데이터 | 프론트 API 키 노출·토큰 비용 문제 |
| 갤러리 구조 | 쇼케이스 탭 + /showcase 독립 페이지 | 탭은 대회 컨텍스트 유지, 독립 페이지는 외부 공유용 |
| AI 요약 고정 | 제출 시 1회 생성 후 localStorage 저장 | 관람자마다 다른 요약 방지, 토큰 절약 |
| 백엔드 | 없음 (SSG + localStorage) | 배포 단순화, 심사자 별도 설정 불필요 |
| 로그인 | 열람은 불필요, 제출만 더미 인증 | 외부 트래픽 유입, 접근성 |
